# EmoLang 直譯器核心程式碼解析

EmoLang 的直譯器架構採用了經典的**「前端編譯與樹狀走訪直譯器 (Tree-Walk Interpreter)」**設計。
整個專案被完美地拆分為 5 個 C 語言模組，運行流程如下：

1. **讀檔載入** (`main.c`)：將 `.emo` 腳本讀入記憶體。
2. **詞法分析** (`lexer.c`)：把原始字串切成一個個有意義的單字 (Token)。
3. **語法分析** (`parser.c`)：把平面的 Token 組合成立體的「抽象語法樹 (AST)」，建立邏輯關聯。
4. **執行與求值** (`eval.c`)：走訪 AST、管理虛擬記憶體與變數，並真正執行加減乘除與控制流。

以下是每個模組的詳細功能解析：

---

## 1. `emolang.h` (核心定義與標頭檔)
這是整個專案的「大腦藍圖」，定義了所有模組共用的資料結構與介面。

* **`TokenType` (標記型別)**：定義了 EmoLang 認識的所有字彙編號。包含所有的 Emoji 關鍵字（如 `TOK_LET`, `TOK_IF`, `TOK_AND`）以及基礎符號（如 `TOK_LPAREN` 左括號）。
* **`Value` (動態型別結構)**：這是 EmoLang 能支援**「動態型別」**的核心！它是一個 C 結構體，裡面包含：
  * `type`：標記目前的資料是 (0: 整數/布林, 1: 小數, 2: 字串)。
  * `i`, `f`, `s`：分別用來儲存整數、浮點數與字串資料。
* **`ASTNode` (抽象語法樹節點)**：直譯器的神經元。每個節點代表一個操作（例如加法、If 判斷、宣告變數），它包含了：
  * `left`, `right`：左右子節點（用於數學運算或條件判斷）。
  * `true_branch`, `false_branch`, `body`：控制流需要的分支指標。

---

## 2. `lexer.c` (詞法分析器 Lexer)
負責**「切字串」**。它不會管程式碼的邏輯對不對，它的工作只是無情地把原始碼切成一個個電腦懂的標記 (Token)。

* **`emoji_keywords` 與 `match_keyword()`**：預先建立了一個 Emoji 字典。程式會用 C 語言的 `strncmp` 去比對目前的 UTF-8 字串是否符合我們的保留字（如 `📦`, `🛠️`, `🔗`），並轉換成對應的 Token。
* **`advance_token()`**：核心狀態機。它會略過空白字元，並判斷接下來讀到的是什麼：
  * 如果是 `"..."`，打包成字串 `TOK_STR`。
  * 如果是數字 (包含小數點)，打包成 `TOK_NUM` 或 `TOK_FLOAT_NUM`。
  * 如果是 Emoji，轉換為特定的操作符。
  * 如果都不是，就視為一般的變數名稱 (`TOK_ID`)。
* **`eat(TokenType)`**：語法檢查器。確保目前的 Token 符合預期，並強制往下讀取。如果遇到不合語法的東西（例如本來預期是變數名，卻出現了加號），就會在這裡報錯並中止程式。

---

## 3. `parser.c` (語法分析器 Parser)
負責**「理解邏輯與層級關係」**。它向 Lexer 不斷索取 Token，然後把平面的 Token 陣列組合成立體的「抽象語法樹 (AST)」。採用的是**「遞迴下降解析 (Recursive Descent Parsing)」**。

* **優先級解析鍊 (Operator Precedence)**：
  為了解決「先乘除後加減」與「邏輯運算優先級」的問題，運算被嚴格分層（越底層越先計算）：
  1. `parse_primary`：處理最基礎的數字、字串、變數、括號 `()`，與**函數呼叫 (`變數()`)**。
  2. `parse_postfix`：處理後綴操作，如結構體存取 `➡️`、陣列索引 `📌`。
  3. `parse_prefix`：處理前綴操作，如配置記憶體 `🆕`、取指標 `📍`、解指標 `🎯`、邏輯反轉 `🙅`。
  4. `parse_factor` & `parse_addition`：處理乘除餘數 (`✖️`, `➗`, `✂️`) 與 加減法 (`➕`, `➖`)。
  5. `parse_comparison`：處理比較大小 (`🤝`, `📈`, `📉`)。
  6. `parse_logical_and` & `parse_expression`：處理邏輯 AND (`🔗`) 與 OR (`🔀`)。
* **`parse_statement()`**：解析每一行的敘述句。
  * 例如遇到 `🤔` (If)，它就會把接下來的表達式設為條件，並呼叫 `parse_block()` 去讀取 `👇` 到 `👆` 之間的所有指令。
  * 這裡也巧妙地處理了 `🤷🤔` (Else If) 的遞迴串接邏輯。
  * 處理了函數定義 `🛠️` 與回傳 `🔙`。
* **`parse_block()`**：專門用來處理 `👇 ... 👆` 之間的區塊，把多行指令串成一個單向的鍊表 (`stmt->next`)。

## 4. `eval.c` (執行環境與求值器 Evaluator)
這部分是整個直譯器的**「大腦與記憶體」**。語法樹建好後，就由它來跑透透並產生真實結果。

* **虛擬記憶體 (`memory` / `alloc_mem`)**：
  建立了一個龐大的 `Value memory[10000]` 陣列。當我們使用 `🆕` 建立陣列或結構體時，直譯器會從這個陣列中切割一塊連續的格子出來分配給物件，完美模擬了 C 語言的 Heap 記憶體分配。
* **呼叫堆疊與區域變數 (`sym_stack` / `call_depth`)**：
  為了支援函數的「區域變數隔離」，實作了 Scope 機制。
  `call_depth` 記錄目前的函數深度（全域是 0）。當呼叫函數時，`call_depth` 加 1，並在此層級註冊參數；函數結束後 `call_depth` 減 1，區域變數自動失效，避免變數名稱污染。
* **`eval(ASTNode)` (求值引擎)**：
  負責把 AST 節點轉換成實質的 `Value` 數值。
  * 支援動態型別：字串與數字的安全拼接 (利用 `snprintf` 與 `strncat`)。
  * 支援**短路求值 (Short-circuiting)**：當遇到 AND (`🔗`) 且左邊為 False 時，直接回傳 False，不再浪費效能去計算右邊；OR (`🔀`) 亦然。
  * 當遇到函數呼叫 `AST_FUNC_CALL`，會啟動呼叫堆疊機制，並執行該函數的 AST 樹狀結構。
* **`execute(ASTNode)` (執行控制器)**：
  依序走訪每一行 Statement。
  * 負責變數賦值 (`AST_ASSIGN`)、印出畫面 (`AST_PRINT`)。
  * 負責執行迴圈：把 EmoLang 的 `🔁` 與 `🎡` 轉換成 C 語言底層的 `while` 來執行。
  * **中斷機制**：包含一個關鍵的全域變數 `is_returning`。當執行到 `🔙` 時會設為 1，並立刻 `return` 中斷目前的 C 語言執行流程，達成跳出函數並攜帶回傳值的效果。

---

## 5. `main.c` (主程式進入點)
這是最外層的包裝，負責啟動整個編譯與執行流程。

* **讀檔**：透過 `fopen` 跟 `fread` 將 `.emo` 腳本檔案的所有文字 (包含 UTF-8 編碼的 Emoji) 一次性讀進 `src_code` 字元陣列中。
* **建構語法樹**：
  呼叫 `advance_token()` 讓 Lexer 準備好第一個標記。
  在 `while` 迴圈中不斷呼叫 `parse_statement()`，直到檔案結束 (`TOK_EOF`)，藉此建立整支程式的 AST。
* **啟動執行**：
  把建立好的 AST 根節點丟給 `execute(program)`，程式開始生龍活虎地運行，並把結果印到終端機上。最後透過 `free` 釋放源碼佔用的系統記憶體。
