# EmoLang 直譯器核心程式碼解析 (v6.0 裸機進化版)

EmoLang 的直譯器採用了經典的**「前端編譯與樹狀走訪直譯 (Tree-Walk Interpreter)」**設計。最特別的是，它是一套**零相依 (Zero-Dependency)** 的系統級軟體 —— 完全不依賴 C 語言的 `<stdio.h>` 或 `<stdlib.h>`，全靠直接與作業系統 (OS) 溝通來運作！

整個專案拆分為 **3 個標頭檔 (Headers)** 與 **7 個原始碼模組 (Source Files)**。

---

## 📂 1. 定義與介面 (include/ 資料夾)

### `emolang.h` (核心藍圖)
整個專案的神經中樞。這裡定義了 EmoLang 的基因：
* **`TokenType` 與 `ASTType`**：列出所有 Emoji 對應的標記編號與語法樹節點類型。
* **`Value` (動態型別核心)**：讓 EmoLang 具備類似 Python 的動態型別能力。它是一個結構體，包含 `type` (0:整數/布林, 1:小數, 2:字串, 3:列表, 4:字典) 以及存儲資料的 `i`, `f`, `s` 欄位。
* **`extern` 全域宣告**：用來把分散在各個 `.c` 檔的變數（如 `src_code`、`memory`）牽線綁定在一起。

### `io.h` & `utils.h` (底層工具介面)
* 取代標準函式庫的介面。`utils.h` 甚至使用了 C 語言的「巨集魔法 (`#define strcmp my_strcmp`)」，讓編譯器自動把舊的標準庫呼叫，無縫替換成我們自己手寫的函式。

## 📂 2. 底層引擎 (src/io.c & src/utils.c)

這個區塊是 EmoLang 能脫離 C 標準庫獨立運作的秘密武器。

### `io.c` (作業系統 I/O 通訊)
不使用 `printf` 或 `scanf`，而是直接呼叫 POSIX 作業系統的底層 System Calls：
* **`write(1, ...)`**：向檔案代號 1 (標準輸出 stdout，也就是螢幕) 寫入位元組，用來實作 `my_print_str` 等輸出功能。
* **`read(0, ...)`**：從檔案代號 0 (標準輸入 stdin，也就是鍵盤) 讀取位元組，實作 `my_read_str` 來接收使用者的 `📥` 輸入。
* **`my_open / my_read`**：直接開啟硬碟檔案，把 `.emo` 腳本讀入記憶體。
* **`my_error()`**：統一的崩潰處理器，印出錯誤訊息後發送 `_exit(1)` 中斷程式。

### `utils.c` (徒手刻出的字串處理器)
* **字串操作**：自己用 `while` 迴圈與指標偏移，刻出了 `my_strlen`, `my_strcmp`, `my_strncat` 等功能。
* **型別轉換 (`my_itoa`, `my_ftoa`, `my_atoi`)**：把字串 `"123"` 透過 ASCII 碼運算 (`c - '0'`) 轉成電腦底層的二進位數字，或是反過來把數字轉成字串，以便印到螢幕上。

## 📂 3. 前端解析 (src/lexer.c & src/parser.c)

這個區塊負責把人類寫的 Emoji 文字，轉化為電腦能懂的立體結構。

### `lexer.c` (詞法分析器 / 切詞器)
* **運作邏輯**：逐字元讀取 `src_code`。它會略過空白，並把連續的字串 (`"..."`)、數字 (`123.4`) 或 Emoji 關鍵字 (`📦`, `🤔`) 打包成一個個的 `Token`。
* **字典比對**：利用 `emoji_keywords` 陣列，透過比對 UTF-8 位元組來判斷目前讀到的 Emoji 代表什麼 Token。

### `parser.c` (語法分析器 / 建樹器)
採用「遞迴下降解析 (Recursive Descent Parsing)」來處理「先乘除後加減」與優先級。
* **優先級分層**：從最外層的 `parse_expression` (處理邏輯 OR `🔀`)，一路往內遞迴到 `parse_primary` (處理數字、變數或括號 `()`)。越底層的函數，在數學上的優先級越高。
* **控制流解析 (`parse_statement`)**：遇到 `🤔` (If) 或 `🔁` (While) 時，會去解析條件，接著呼叫 `parse_block()` 去把 `👇` 到 `👆` 之間的區塊打包成一個分支 (Branch) 掛在樹上。

## 📂 4. 記憶體管理 (src/ast.c & src/memory.c)

### `ast.c` (靜態語法樹池)
* 因為我們不能使用 `malloc`，所以這裡宣告了一個超大陣列 `ASTNode ast_pool[50000];`。
* `create_node()` 每次被呼叫時，就從這個大陣列裡發配一個空的節點出去。這種做法稱為「靜態記憶體池 (Static Memory Pool)」，效能極高。

### `memory.c` (虛擬機器記憶體與狀態)
EmoLang 虛擬機的核心儲存區：
* **變數儲存 (`memory[10000]`)**：這是一塊連續的虛擬 Heap，透過移動 `heap_ptr` 來分配變數的空間。
* **動態結構池 (`list_pool`, `dict_pool`)**：專門用來存放 `📋` (列表) 和 `📖` (字典) 的資料。
* **呼叫堆疊 (`sym_stack`, `call_depth`)**：
  用來實作**「函數的區域變數」**！每當呼叫一個函數，`call_depth` 就加 1，所有新變數都會存在新的一層。函數結束後深度減 1，那一層的變數就自動對外隱藏，完美實現了 Scope 隔離！

## 📂 5. 執行引擎與進入點 (src/eval.c & src/main.c)

### `eval.c` (求值與執行引擎)
這顆大腦負責「走訪 (Traverse)」整棵語法樹並執行動作：
* **`eval(ASTNode *node)` (求值器)**：
  遞迴計算節點的值。包含加減乘除、字串串接、字典讀取。
  這裡實作了**「短路求值 (Short-circuiting)」**：遇到 `🔗` (AND) 且左邊為 False 時，直接回傳 False，不計算右邊。
* **`assign_value()` (賦值器)**：
  統一處理變數寫入、指標寫入 `🎯`、陣列寫入 `📌`，甚至是字典的動態 Key 新增。
* **`execute(ASTNode *stmt)` (執行控制器)**：
  依序走訪每一行 Statement，呼叫 `eval` 運算，並利用 C 語言底層的 `while` 和 `if` 來實作 EmoLang 的迴圈。它也負責捕捉 `🔙` (Return) 觸發的 `is_returning` 中斷信號，用來跳脫函數。

### `main.c` (主程式進入點)
* 不使用 `fopen`，而是用 OS 的 `my_open` 與 `my_read`，直接將 `.emo` 腳本讀入我們預先分配好的 1MB `src_buffer` 陣列中。
* 呼叫 Parser 建構 AST，最後將 AST 的根節點 (Root Node) 交給 `execute()`，正式點火啟動整支 EmoLang 程式！
