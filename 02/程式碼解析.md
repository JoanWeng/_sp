# EmoLang 直譯器核心程式碼解析 (v4.0)

EmoLang 的直譯器架構採用了經典的**「前端編譯與樹狀走訪 (Tree-Walk Interpreter)」**設計。整個專案被拆分為 5 個模組，運行流程如下：

1. **讀取源碼** (`main.c`)：讀取 `.emo` 檔案。
2. **詞法分析** (`lexer.c`)：把字串切成一個個的 Token（標記）。
3. **語法分析** (`parser.c`)：把 Token 組合成 AST（抽象語法樹）。
4. **執行與求值** (`eval.c`)：走訪 AST 並管理記憶體與變數，產生最終結果。

以下是每個模組的詳細功能解析：

---

## 1. `emolang.h` (核心定義與標頭檔)
這是整個專案的「藍圖」，定義了所有模組共用的資料結構與介面。

* **`TokenType` (標記型別)**：定義了 EmoLang 認識的所有字彙（例如 `TOK_LET`, `TOK_IF`, `TOK_PLUS`）。
* **`Value` (動態型別結構)**：這是 EmoLang 能支援動態型別的核心！它是一個 C 結構體，裡面包含了 `type` (標記目前是整數、小數還是字串)，以及 `i`, `f`, `s` 來分別儲存對應的數值。
* **`ASTNode` (抽象語法樹節點)**：直譯器的神經元。每個節點代表一個操作（例如加法、If 判斷、宣告變數），它包含了左右子節點指標 (`left`, `right`)，以及控制流需要的分支指標 (`true_branch`, `body` 等)。

---

## 2. `lexer.c` (詞法分析器)
負責**「切字串」**。它不會管程式碼的邏輯對不對，它的工作只是無情地把原始碼切成一個個電腦懂的單字。

* **`emoji_keywords` 與 `match_keyword()`**：預先建立了一個 Emoji 字典。當程式讀取時，會用 C 語言的 `strncmp` 去比對目前的 UTF-8 字串是否符合我們的保留字（如 `📦`, `🛠️`, `🤔`）。
* **`advance_token()`**：核心函式。它會略過空白字元，並判斷接下來是一段字串 (`"..."`)、一串數字 (整數或浮點數)、一個預設的 Emoji 關鍵字，還是一般的變數名稱 (`ID`)，最後將結果存入全域變數 `current_token`。
* **`eat(TokenType)`**：確保目前的 Token 符合預期，並強制往下讀取下一個 Token。如果遇到不合語法的東西（例如本來預期是變數名，卻出現了加號），就會在這裡報錯並結束程式。

---

## 3. `parser.c` (語法分析器)
負責**「理解邏輯與層級關係」**。它會向 Lexer 不斷要 Token，然後把平面的 Token 陣列組合成立體的「樹 (AST)」。採用的是標準的 **「遞迴下降解析 (Recursive Descent Parsing)」**。

* **優先級解析鍊 (`parse_expression` 到底層)**：
  為了解決「先乘除後加減」的問題，它將運算分層：
  1. `parse_primary()`：處理最基礎的數字、字串、變數、括號 `()`，與**函數呼叫**。
  2. `parse_postfix()`：處理後綴操作，如結構體存取 `➡️`、陣列索引 `📌`。
  3. `parse_prefix()`：處理前綴操作，如配置記憶體 `🆕`、取指標 `📍`、解指標 `🎯`。
  4. `parse_factor()`：處理乘除餘數 (`✖️`, `➗`, `✂️`)。
  5. `parse_addition()`：處理加減法 (`➕`, `➖`)。
  *每一層都會呼叫下一層，確保括號和乘除法會在 AST 樹的最底端（優先被執行）。*
* **`parse_statement()`**：解析每一行的敘述句。例如遇到 `🤔` (If)，它就會把接下來的表達式設為條件，並呼叫 `parse_block()` 去讀取 `👇` 到 `👆` 之間的所有指令。這裡也處理了 `🤷🤔` (Else If) 的遞迴串接邏輯。
* **`parse_block()`**：專門用來處理 `👇 ... 👆` 之間的區塊，它會把多行指令串成一個單向的鍊表 (`stmt->next`)。

---

## 4. `eval.c` (執行環境與記憶體)
這部分是整個直譯器的**「大腦與記憶體」**。語法樹建好後，就由它來跑透透。

* **記憶體與虛擬 Heap (`memory` / `alloc_mem`)**：
  建立了一個龐大的 `Value memory[10000]` 陣列。當我們使用 `🆕` (new) 建立陣列或結構體時，直譯器會從這個陣列中切割一塊連續空間出來分配給物件。
* **呼叫堆疊與區域變數 (`sym_stack` / `call_depth`)**：
  為了支援函數的「區域變數」，這裡實作了 Scope 機制。
  `call_depth` 記錄目前的函數深度。全域是第 0 層。當我們呼叫函數時，`call_depth` 加 1，並在此層級註冊參數。函數結束後 `call_depth` 減 1，區域變數就會自動失效，完美隔離了外界！
* **`eval(ASTNode)` (求值器)**：
  負責把節點轉換成實質的 `Value`。如果是數字就回傳數字；如果是 `AST_BINOP` (運算)，就會遞迴計算左右兩邊的值再相加。這裡也實作了字串與數字的安全拼接 (`snprintf`/`strncat`)。
  當遇到 `AST_FUNC_CALL` 時，會啟動上述的 `call_depth` 機制，並執行該函數的語法樹。
* **`execute(ASTNode)` (執行器)**：
  依序執行 Statement。如果遇到 `AST_LET`，就幫變數分配記憶體。如果遇到 `AST_WHILE`，就用 C 語言的 `while` 迴圈來不斷觸發 `execute()`。
  這裡還有一個關鍵變數 `is_returning`：當遇到 `🔙` 時會設為 1，並立刻 `return` 中斷目前的 C 迴圈，達成跳出函數的效果。

---

## 5. `main.c` (主程式進入點)
這是最外層的包裝，負責串接所有模組。

* **讀檔**：透過 `fopen` 跟 `fread` 將 `.emo` 腳本檔案的所有文字 (包含 Emoji) 一次性讀進 `src_code` 字元陣列中。
* **啟動編譯**：
  呼叫 `advance_token()` 讓 Lexer 準備好第一個標記。
  在 `while` 迴圈中不斷呼叫 `parse_statement()` 建立整支程式的 AST。
* **啟動執行**：
  把建立好的 AST 頭節點丟給 `execute(program)`，程式開始運行，並把結果印到終端機上。最後透過 `free` 釋放源碼佔用的系統記憶體。
